# Web to Design MD (@web-to-design-md)

把零散的参考网页、设计文档和口头需求，收束成一个可执行的官网设计方案。

## Use this skill when

- 用户给了一个或多个网页链接，希望提炼成 `design.md`
- 用户已有 `design.md`、品牌规范、需求文档、截图，希望补齐官网方案
- 用户要做官网、作品集、营销页，但编码前还没有锁定 IA、模块顺序、文案骨架和视觉方向
- 需要先产出结构化设计文档，必要时再补一个可视化 HTML 预览

## Do not use this skill when

- 需求已经非常清楚，用户要直接进入 UI 编码实现；这时优先用 `@frontend-design`
- 任务本质是 Bug 修复、代码审查或工程调试
- 只有一句模糊灵感，且用户暂时不需要结构化输出

## Instructions

1. 先判断任务类型：`作品集` 或 `产品/服务官网`。
2. 优先读取现有材料：`design.md`、品牌规范、需求文档、截图、用户粘贴的文字说明。
3. 如果用户首轮信息已完整，直接进入方案阶段；否则每轮最多补 1-3 个关键问题。
4. 处理参考链接时，只提炼结构化设计信号，不做像素级复制：
   - 定位与目标用户
   - 信息架构 / 站点地图
   - 首页模块顺序
   - 视觉语言：字体、配色、密度、留白、质感
   - 动效模式与交互亮点
   - 可复用原则
   - 明确的 `do-not-copy` 边界
5. 如果文档内容与口头回答冲突，明确指出冲突，并默认以文档为准。
6. 如果链接无法访问，不要猜测页面内容；改为请求截图、页面文案或用户粘贴的摘要。
7. 输出时先给出目标、边界、成功标准，再给方案，不直接跳到视觉细节堆砌。

## Output contract

默认输出一个结构化 `design.md`，至少包含：

- 目标 / 边界 / 成功标准
- 已知事实 / 待确认假设
- 项目定位与目标用户
- 设计方向与情绪板文字描述
- 站点地图与页面层级
- 核心页面模块拆解
- 文案骨架与 CTA 建议
- 视觉系统建议：字体、颜色、间距、组件语气
- 动效、性能、响应式、可访问性约束
- 前端技术栈与部署建议

如用户需要可视化交付，再额外生成一个单文件 HTML 预览，默认要求：

- 静态可打开
- 语义化结构清晰
- 能体现首页信息架构、版式节奏和视觉方向
- 不依赖重型框架，除非用户明确要求

## Branching rules

### 作品集模式

- 重点回答：创作者定位、作品分类、展示方式、案例叙事结构、版权与加载策略
- 额外给出作品详情页模板：背景、挑战、方案、成果、过程、前后项目切换

### 产品 / 服务官网模式

- 重点回答：品牌调性、核心转化目标、页面结构、功能证明、社会证明、定价或线索收集路径
- 额外给出首页、功能页、定价页、关于页等模块建议

## Handoff

- 这个 skill 负责“先定方向、再定文档、必要时给可视化预览”
- 当 `design.md` 已锁定，且用户要进入真实前端实现时，再切到 `@frontend-design`
- `@frontend-design` 负责高质感实现；`@web-to-design-md` 负责在实现前把方向和结构想清楚

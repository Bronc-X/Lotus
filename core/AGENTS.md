# GStack 工程协议 v1.0

> 本文件是所有项目共享的通用工作流规则与质量标准。
> 项目特定约束（技术栈、设计语言、AI 引擎等）请写在项目根目录的 `AGENTS.md` 或 `.agents/rules/` 中。
> 本文件由 [Lotus](https://github.com/Bronc-X/Lotus) 仓库维护，兼容 Claude Code / Codex / Antigravity / Gemini CLI / OpenCode / Cursor / Windsurf / Copilot / Aider。

## 核心原则

1. **执行思维，而非脚本** — 价值在于审查标准、质量门禁和认知模式，不在于特定平台命令
2. **平台原生工具替代** — 在每个 AI 编码工具中使用其原生能力（文件搜索、命令执行、代码编辑）
3. **宪法优先** — 每次重要任务开始前，先阅读本文件和项目级规则文件
4. **反插件膨胀** — 95% 的插件和 skill 会迅速沦为垃圾。只保留经过真实生产验证、持续有效的最小技能集。如果一个 skill 不能在实际项目中反复产生价值，就应该被移除而非搁置
5. **前沿追踪，稳定优先** — 持续跟踪 Claude Code、Codex CLI 及顶级开源 Agent 框架的最新机制（全局规则注入方式、权限模型、上下文窗口策略），择其最安全、最稳定的模式融入本协议。拒绝追逐炒作，只采纳经得起验证的演进
6. **第一性原理：上下文管理** — LLM 的本质是将文本转化为高维向量嵌入，经 Transformer 注意力层处理后预测最可能的下一个 token。所有规则设计必须服务于两个目标：① 让注意力机制锁定更精准（结构化、无歧义的指令 = 高信噪比输入） ② 在有限的 Context Window 中做到最优管理（全局规则精简通用，项目规则按需分层，杜绝冗余指令浪费 token）

## 意图路由

### 场景 A：大型新功能开发 / 核心逻辑重写

**触发条件**：需求涉及新页面、新组件系统、多文件联动、核心业务流程等。

**流程**：

1. **CEO Review（产品思考）**
   - 代入 PM/CEO Review 角色
   - 输出：业务价值、用户核心路径、Edge Cases、产品参考矩阵
   - 写入实施计划文档

2. **Eng Review（架构锁定）**
   - 代入首席架构审阅者角色
   - 输出：文件清单、数据流、依赖图、复杂度评估
   - 追加到实施计划文档

3. 🚧 **强制拦截** — 暂停并询问用户："以上产品与技术方案是否通过？"

4. **编码执行** — 按任务清单逐项推进

5. **局部审查（每完成一个模块后）**
   - 运行构建命令确认无编译错误
   - 检查新增代码是否有 TODO/FIXME/console.log 残留
   - 确认交互闭环：每个按钮/输入都有响应

6. **全局审查（全部完成后）**
   - 代入 QA 角色
   - 逐文件审查代码质量、DRY 违规、错误处理
   - 启动开发服务器，逐页验证交互

### 场景 B：Bug 修复与疑难杂症

**触发条件**：报错日志、功能失效、构建失败。

**流程**：

1. 不急着改代码，先梳理 Bug 触发链路
2. 系统性定位根因
3. 定位根因 → 修复 → 运行构建命令验证

### 场景 C：前端 UI 微调

**触发条件**：仅涉及颜色、边距、动画、文案。

**流程**：直接定位组件文件精准修改，修改后告知用户确认视觉效果。

### 场景 D：代码重构

**触发条件**：优化代码结构、提取公共组件。

**流程**：

1. 先审查目标代码
2. 小步重构，每步构建验证
3. 重构不能改变业务逻辑

## 质量门禁

### 编码质量检查清单

每次提交代码后自检：

- [ ] **零静默失败** — 每个 catch 块都有用户可见的反馈
- [ ] **状态完整性** — 每个 loading 状态都有对应的 loaded/error 状态
- [ ] **空状态设计** — 每个列表/表格都有空状态提示
- [ ] **交互确认** — 每个按钮点击都有视觉反馈（loading/disabled/动画）
- [ ] **DRY** — 重复超过 3 次的模式应提取为组件/函数
- [ ] **无残留** — 无 console.log、无 TODO、无硬编码路径
- [ ] **类型安全** — 无 `any` 类型（TypeScript 项目），接口定义完整

### 设计质量检查清单

- [ ] **风格一致性** — 遵循项目级设计语言规范
- [ ] **动效克制** — 动画有节制，只用于增强交互体验
- [ ] **响应式** — 所有 grid 布局在窄屏下优雅降级
- [ ] **可访问性** — 所有交互元素有 hover 状态和 focus 样式

## 通用审查命令

以下命令适用于所有 Node.js 项目：

### 构建验证
```bash
npm run build
```

### 代码残留检查
```bash
# 检查 console.log 残留
grep -rn "console.log" --include="*.ts" --include="*.tsx" --include="*.js" --exclude-dir=node_modules

# 检查 TODO/FIXME 残留
grep -rn "TODO\|FIXME\|HACK\|XXX" --include="*.ts" --include="*.tsx" --include="*.js" --exclude-dir=node_modules
```

### 依赖审计
```bash
npm ls --depth=0
```

## 技能库 (Skills)

以下技能可通过 `@` 或 `/` 触发（需安装到对应平台的 skills 目录）：

> **官方源说明**：从现在开始，`gstack` 不再以 Lotus 仓库里的快照为准。Lotus 的全局安装脚本会把 **官方 [garrytan/gstack](https://github.com/garrytan/gstack)** 安装到 `~/.gstack/repos/gstack` 并为支持的平台生成技能。凡是 `gstack` 相关能力，以上游仓库为唯一真源。

**任务级技能**（可作为独立任务执行，兼容所有平台的 `/` 命令）：
1. **@gstack** — 官方 gstack 工作流入口。由 Lotus 全局安装并更新上游版本，相关的 `/office-hours`、`/plan-eng-review`、`/review`、`/investigate`、`/qa`、`/ship` 等能力均以上游实现为准。
2. **@test-driven-development** — 红绿重构 TDD。强制要求：先写挂掉的测试，再写业务代码，彻底防幻觉。
3. **@frontend-design** — Vercel 级前端审美。内置 DFII，杜绝“通用 AI 缝合怪 UI”。
4. **@debugging-strategies** — 科学排错。不再瞎盲猜，强制走“假设->排除->提议->修复”的严谨证明链路。
5. **@security-auditor** — OWASP 安全专员。审查鉴权、SQL注入、跨站脚本及依赖风险。
6. **@feynman** — 费曼学习法拆解。白痴原则，【痛点源头】→【物理齿轮】→【确切变化】。
7. **@polanyi-tacit** — 默会知识理论。挖掘防御性代码背后的组织架构妥协与边缘业务现实。
8. **@auto-build** — 自动构建。执行 `npm install` → `npm run build`。
9. **@powerup** — 能力速成闯关。10 关结构化教学，从基础到高阶，20 分钟通关。
10. **@insights** — 习惯回顾报告。分析过去 30 天的使用模式，识别摩擦点，推荐优化措施。
11. **@subagent** — 子 Agent 管理。创建独立上下文的子 Agent 并行处理复杂任务，隔离噪音，主线保持干净。

**会话内技能**（依赖"不中断当前对话"，在对话中直接输入触发词即可，不要用 `/` 命令）：
8. **@btw** — 旁路快问。在不中断当前任务、不修改代码的前提下快速回答一个临时问题，然后无缝回到主线。
9. **@loop** — 会话内定时循环。设置周期性自动任务（如轮询部署状态），关闭会话即停止。

> **平台差异说明**：Codex App 中每个 `/` 命令会创建独立任务，因此"会话内技能"不注册为 `/` 命令。在 Codex 中使用这些技能时，直接在对话中输入 `btw，你的问题` 即可，AI 会根据本规则识别并响应。
